Summary Two

Currently Stacked on: #27534 review the last commit to see changes relevant to this PR exclusively Historically React would produce component stacks for dev builds only. There is a cost to tracking component stacks and given the prod builds try to optimize runtime performance these stacks were left out. More recently React added component stacks to Fiber because it can be immensely helpful in tracking down hard to debug production issues. Fizz was not updated to have a similar behavior With the advent of prerendering however stacks for production in Fizz are more relevant because prerendering is not really a dev-time task. If you want the ability to reason about errors or postpones that happen during a prerender having component stacks to interrogate is helpful and these component stacks need to be available in production otherwise you are really never going to see them. (it is possible that you could do dev-mode prerenders but we don't expect this to be a common dev mode workflow) To better support the prerender use case and to make error logging in Fizz more useful the following changes have been made onPostpone now accepts a second postponeInfo argument which may contain a componentStack. Postpones always originate from a component render so the stack should be consistently available in the following cases In production it will contain a stack during prerendering In dev it will contain a stack always onError now accepts a second errorInfo argument which may contain a componentStack. If an error originated from a component a stack will be included in the following cases. In production it will contain a stack during prerendering In dev it will contain a stack always This change entails tracking the component hierarchy in prod builds now. We don't actually reify a stack string unless we're in a mode which produces stacks (as described above). The hope is that the incremental cost of stack tracking in prod is not significant if we just use a linked list style stack and only traverse it when we must actually produce something for the above APIs It's possible that adding stacks to these APIs in prod all the time is reasonable since the work only happens when you are in an error pathway where performance is already impacted. We can decide to modify this PR to accommodate that with discussion or land that as a separate change if we decide it is worth adding In the course of implementing this change a number of simplifications were made to the code which should make the stack tracking more resilient. We no longer use a module global to curry the stack up to some handler. This was delicate because you needed to always reset it properly. We now curry the stack on the task itself. This still requires some careful code coordination because we want to always "consume" this stack at the point where we are handling the thrown value. Another change made was to track the component stack on SuspenseBoundary instances so that we can provide the stack when aborting suspense boundaries to help you determine which ones were affected by an abort.
